## 名字与初衷

**Lacuna**（拉丁语）：空白、缺口、间隙。

传统视觉小说面临一个老问题：作者写得越细，玩家选择的真实重量越轻——大量分支变成假分支。但如果让 LLM 自由生成，又会出现"剧情失控"：泄露不该泄露的真相、违背角色硬设定、把伏笔写散。

Lacuna 想做的折中是：**作者主动留白，LLM 填补留白，玩家选择决定填法**。

## 三层架构

| 层 | 职责 |
|---|---|
| **Canon（结构化层）** | 角色硬设定、世界观、玩家累计状态、关系数值——每次必带，硬事实绝对优先 |
| **Memory（语义层）** | 场景过往的软描写、伏笔——hybrid 召回 = 关键词重叠 × 时间衰减 × 视角可见性 |
| **Scene Contract（场景契约）** | 作者写的节点骨架——必须发生、不可揭露、NPC 知情边界、关系驱动的语气 |

LLM 不决定故事走向、不能揭露秘密、不能改写硬事实，**只在作者预设的"受控空白"里写细节、对白、变体**。

## 实测效果

带 demo《雨后旧校舍》。同一个玩家在第一节选了"直接追问真由"（触发 `mayu→player.trust −5`），到第二节门口时 LLM 自发写出：

> 真由站在台阶下面那一截，没有再往上走。**先前在教室里，你当着所有人追问过她；她现在的眼神还留着那时候的刺。**
>
> "你真的要进去。"她的声音被湿风削薄了，"我说过的话，你一句都没听。"

**加粗部分是 LLM 自己织进去的**——作者没在第二节脚本里指定要回响第一节的选择，是关系数值差让模型自动调整了真由的态度和措辞。

到第四节天台时，LLM 一次性回响了三个具体过往选择，并主动写出元叙事点题："**她记得。每一步她都记得。**"

> 这两段证明的不是"LLM 写得好"——任何强模型都能写文学性文本。它们证明的是引擎架构成立：作者只写了 must_establish + npcs.knows + tone_options + canon_update.relations 这几个结构化字段，LLM 就能在严格不泄露真相的前提下，让玩家感觉「她记得我」。

## 工程上的几个权衡

- **流式输出 vs JSON schema 严格约束**：选了流式（玩家 1-2 秒看到第一个字），靠 system prompt 约定 + Opus 4.7 指令遵循
- **must_establish 校验改成异步**：漏事实时只 `[警告]` 不阻塞玩家
- **不再 retry**：流式打出去的文本无法收回
- **content / engine 解耦**：改 YAML 就改了故事，引擎一行不动；改引擎，content 不用动——作者和工程师可以并行迭代
